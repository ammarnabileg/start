# UX-AUDIT — التوثيق البصري الكامل للنظام

دليل بصري شامل لكل شاشات منصّة الولاء (تطبيق العميل + تطبيق التاجر)، مرتّب حسب
التسلسل الفعلي لرحلة المستخدم، بحيث يمكن فهم المنتج بالكامل من الصور وحدها.

## المنهجية والنطاق

- **Full View — بدون قص:** كل صورة مقاس `390×844` (هاتف بورتريه) وتُظهر كل العناصر
  الثابتة كما يراها المستخدم: الـHeader، الـBottom Navigation (بالزر البارز
  وشارة الإشعارات)، أشرطة الحالة، والأزرار الثابتة. لا يوجد أي Component مقصوص.
- **مصدر الصور:** تُولَّد من حزم لقطات الشاشة (Flutter screenshot harnesses) التي
  تبني الشاشات من **نفس** design-system و widgets المستخدمة في التطبيق الحقيقي
  (`loyalty_core`)، بخط Tajawal واتجاه RTL — فتطابق ما يراه المستخدم.
- **التقارير مبنية على الكود الفعلي:** كل حقل في تقرير الشاشة (Route, Actions,
  Bugs…) مستخرَج من قراءة الكود المصدري الحقيقي (router، الشاشات، المعالجات).

## حدود هذا التدقيق (شفافية)

لا يوجد في بيئة التنفيذ مُحاكٍ/جهاز حيّ ولا backend مباشر، لذلك:

- لا يمكن التقاط **Console Errors / API Errors وقت التشغيل** فعليًا — استُعيض عنها
  بتحليل ساكن (Static Analysis) وقراءة الكود لاكتشاف المخاطر.
- بعض الحالات الديناميكية (نتيجة API فاشلة فعلية، Responsive على أحجام مختلفة)
  مُوثّقة كملاحظات مبنية على الكود لا كلقطة حيّة.

## أبرز نتائج الجودة (ملخّص تنفيذي)

### تطبيق العميل
- **زرّ ميّت:** «تفعيل» في `LocationPrimingScreen` (TODO stub) — لا يطلب الإذن فعليًا.
- **قفز لتبويب خاطئ:** «استبدل نقاطك» يثبّت `animateTo(3)` بينما التبويبات شرطية.
- **تطابق اللقطة ↔ الكود (صُحِّح هنا):** الدخول كان OTP→أصبح هاتف+كلمة مرور؛ OTP كان 4 خانات/0:42→6 خانات/0:30.
- مخاطر null-safety: `currentUserId!`، `DateTime.parse` بلا حماية (كوبون/استبدال).
- Profile حالة الخطأ بلا retry؛ Notifications بلا تفاعل + أثر جانبي في بناء المزوّد.

### تطبيق التاجر
- **ثغرات صلاحيات:** بوّابة الـHub **fail-open**؛ ردّ التقييمات بلا بوّابة؛ برنامج الإحالة يُحفَظ بـ`view` فقط؛
  Dashboard/Profile تتجاوزان البوّابة.
- **إجراء مزدوج:** الماسح بلا تهدئة بعد النجاح؛ `record-visit` بلا idempotency؛ `_resend` بلا `_busy`.
- **تحطّم محتمل:** `DateTime.parse` بلا حماية (الاشتراك/الملف)؛ تحويلات JSON ديناميكية؛ `Dropdown.value` خارج العناصر.
- **بلا تأكيد لعمليات خطيرة:** بثّ إعلان لكل العملاء، إبطال مفتاح POS.
- **تسرّبات/سلامة بيانات:** controllers عجلة الحظ/POS بلا dispose؛ تبدّل معرّفات خيارات الأسئلة يُيتّم العدّادات.
- **لا أزرار ميّتة ولا تنقّل مكسور** في تطبيق التاجر.

> القائمة الكاملة المرتّبة بالأولوية + مواضع الأسطر في `REPORT.md` بكل مجلد.

## ملاحظة منهجية
لقطات `m##`/`c##` **بدائل ساكنة (facsimiles)** تُبنى من نفس design-system؛ تمثّل الشكل بدقّة لكنها لا
تُظهر تلقائيًا الحالات الديناميكية (فارغ/خطأ/بوّابات fail-open). التقارير تعتمد ملفات الميزات الحقيقية
كمصدر-حقيقة. أُضيفت لقطات الشاشات الأربع العميقة في تطبيق التاجر (ReportChat، StaffMessages،
ReferralProgram، SetupChecklist) فأصبح المجموع: العميل 41 شاشة، التاجر 45 شاشة.

### ما لم يُغطَّ بعد (لطلب لاحق)
**متغيّرات الحالة المنفصلة** لكل شاشة (تحقّق نموذج خاطئ/صحيح، فارغ، خطأ+إعادة، كل Modal/Dropdown
مفتوح على حدة) لم تُولَّد كلقطات مستقلة — فهي مئات اللقطات الإضافية. وُثّقت هذه الحالات نصيًّا في
`REPORT.md` (حقول States/Forms/Modals). أخبرني إن أردت توليدها بصريًّا واحدة-واحدة.
